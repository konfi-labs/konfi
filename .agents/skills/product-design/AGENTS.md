# Product Design Skill Instructions

- Keep `SKILL.md` as the routing and operating contract. Put detailed standards
  in `references/`.
- Add product-design rules only with current-source evidence and human
  acceptance.
- Give durable rules stable IDs such as `rule/destructive-action-labels`.
- Record scope, rationale, source, exceptions, and bad/good examples when
  promoting a decision into guidance.
- Prefer a linter or deterministic check only when code can identify the failure
  reliably and the fix is concrete.
- Use `references/coverage-gaps.md` for missing standards, disputed guidance, and
  candidate checks that are not accepted yet.
